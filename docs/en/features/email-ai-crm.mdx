# Email AI-CRM

Ring Platform ships an **AI-assisted email CRM** for public contact mailboxes (for example `info@example.com`). Inbound mail is classified, threaded, and optionally answered with Claude; admins review drafts, manage contacts, and track API cost from the admin UI.

> **Architecture:** [Email AI-CRM architecture](/docs/architecture/email-ai-crm) · **API:** [Email AI-CRM API](/docs/api/email-ai-crm) · **Setup:** [Developer guide](/docs/examples/email-ai-crm) · **Migrations:** [Getting started: migrations](/docs/getting-started/migrations)

## Overview

| Layer | Responsibility |
|-------|----------------|
| **Ingestion** | IMAP poll cron (`/api/cron/email-processor`) or inbound webhook |
| **Processor** | `services/email/email-processor.ts` — parse, security, AI, CRM, drafts |
| **Persistence** | PostgreSQL JSONB (`email_threads`, `email_contacts`, `email_messages`, `email_drafts`, `email_tasks`, `email_api_usage`) |
| **Outbound** | `nodemailer` via `EmailSenderService` — replies with `In-Reply-To` / `References` |
| **Admin UI** | `/admin/email-inbox`, drafts, contacts, analytics, tasks, thread detail |

### Business rules

- **MessageId dedup** — same RFC `Message-ID` is never processed twice (avoids duplicate Anthropic spend).
- **Auto-send off by default** — `EMAIL_AUTO_SEND_ENABLED` must be `true` after SMTP smoke test.
- **Cron poll, not IDLE in serverless** — production uses `poll` every 1–5 minutes; optional `EMAIL_PROCESSOR_AUTOSTART` for long-lived Node workers only.
- **Admin-only APIs** — `auth()` + `isPlatformAdmin` on all `/api/admin/email/*` routes.

## Admin routes

| Route | Purpose |
|-------|---------|
| `/{locale}/admin/email-inbox` | Thread list (live API) |
| `/{locale}/admin/email-inbox/[id]` | Thread detail + messages |
| `/{locale}/admin/email-drafts` | Pending AI drafts — approve / reject / send |
| `/{locale}/admin/email-contacts` | CRM contacts registry |
| `/{locale}/admin/email-analytics` | Volume, intent/sentiment, API cost |
| `/{locale}/admin/email-tasks` | Follow-ups and escalations |

Navigation: **Admin → Email Inbox / Drafts / Tasks** (right-rail quick nav when using `AdminWrapper`).

## Processing pipeline

1. **Parse** — headers, body, attachments, thread keys (`References`, `In-Reply-To`).
2. **Security** — 4-layer prompt-injection defense (`services/email/security/`).
3. **AI** — Haiku intent + sentiment; Sonnet response generation with prompt caching.
4. **CRM** — contact upsert, thread upsert, optional auto-tasks.
5. **Draft** — pending queue unless auto-send criteria met.
6. **Notify** — `EmailNotificationService` (in-app / optional Slack) on draft, task, security block, urgent mail.

## Local configuration

Add to **`.env.local`** (see also `env.local.template` **EMAIL CRM** block and [Environment configuration](/docs/deployment/environment#email-ai-crm)):

```bash
# .env.local
IMAP_HOST=mail.example.com
IMAP_PORT=993
IMAP_TLS=true
IMAP_USER=info@example.com
IMAP_PASSWORD=

SMTP_HOST=mail.example.com
SMTP_PORT=587
SMTP_SECURE=false
SMTP_USER=info@example.com
SMTP_PASSWORD=

ANTHROPIC_API_KEY=

CRON_SECRET=generate-a-long-random-string
EMAIL_AUTO_SEND_ENABLED=false
EMAIL_CRM_PERSISTENCE=memory   # omit for Postgres JSONB in integrated dev
```

| Variable | Purpose |
|----------|---------|
| `IMAP_*` / `SMTP_*` | Your community inbox (any IMAP/SMTP provider) |
| `ANTHROPIC_API_KEY` | Intent, sentiment, and draft generation |
| `CRON_SECRET` | Protects `/api/cron/email-processor` when polling |
| `EMAIL_AUTO_SEND_ENABLED` | Keep `false` until SMTP smoke test passes |
| `WEBHOOK_EMAIL_SECRET` | Optional inbound webhook instead of IMAP poll |
| `EMAIL_CRM_PERSISTENCE` | `memory` for quick local UI tests; omit when using migrations `009`/`010` |

## Production deployment

On k8s or Vercel cron, set the same variables from your secret store, apply migrations `009` + `010`, and schedule `POST /api/cron/email-processor` with `{"action":"poll"}` every 1–5 minutes. See [developer guide](/docs/examples/email-ai-crm).
