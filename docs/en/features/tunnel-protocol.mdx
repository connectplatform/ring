# Tunnel Protocol

Ring Platform's realtime layer for per-user inbox updates (notifications, balances) and topic channels (chat, discovery, matcher). The server runs a **custom Node entrypoint** (`server.ts`) that serves Next.js and, when `RING_DEPLOY_TARGET` is `k8s` or `self-hosted`, attaches **native WSS** at `/api/tunnel/ws`. Vercel deployments use serverless route handlers with **SSE + long-polling only**.

Pair with [Push notifications (FCM)](/en/docs/features/push-notifications-fcm) when the browser tab is closed or the tunnel is offline â€” `lib/tunnel/disconnected-tunnel-context.ts` keeps the UI usable while disconnected.

## Architecture

```
server.ts â†’ Next request handler + attachTunnelWss (k8s/self-hosted)
Server services â†’ lib/tunnel/publisher.ts â†’ getTunnelHub() â†’ InMemoryTunnelHub
Next.js routes (/api/tunnel/*) â†’ getTunnelHub()
Client â†’ TunnelProvider / useTunnel â†’ TransportManager â†’ WSS | SSE | poll
```

| Layer | Module | Role |
|-------|--------|------|
| **Broker** | `lib/tunnel/hub/` | `getTunnelHub()`, unified SSE + poll registries, channel subscriptions |
| **Publish API** | `lib/tunnel/publisher.ts` | `publishToChannel`, `publishToUserTunnel`, `broadcastToAll` |
| **Client transport** | `lib/tunnel/transport-manager.ts` | Provider selection, fallback chain, reconnect |
| **React integration** | `components/providers/tunnel-provider.tsx` | Shared connection, progressive connect via `tunnelTimingManager` |
| **Data hooks** | `hooks/use-tunnel.ts`, `hooks/use-sync.ts` | Subscribe/publish; REST fetch + tunnel-driven refresh |

### API routes

| Route | Method | Purpose |
|-------|--------|---------|
| `/api/tunnel/sse` | GET | SSE stream; registers user with TunnelHub |
| `/api/tunnel/ws` | WS | Native WebSocket (k8s/self-hosted custom server only) |
| `/api/tunnel/token` | POST | JWT for tunnel auth (SSE + WSS) |
| `/api/tunnel/poll` | GET/POST | Long-polling inbox for restrictive networks |
| `/api/tunnel/subscribe` | POST | Channel subscription |
| `/api/tunnel/unsubscribe` | POST | Channel unsubscription |
| `/api/tunnel/publish` | POST | Authenticated publish (admin/tools) |
| `/api/tunnel/test` | GET | Environment and provider diagnostics |

## Server publishing

Use **`publishToChannel`** for topic fan-out (chat, discovery, matcher):

```typescript
import { publishToChannel } from '@/lib/tunnel/publisher'

await publishToChannel(`conversation:${conversationId}`, 'message:new', message)
await publishToChannel('opportunities', 'opportunity:created', { id })
```

Use **`publishToUserTunnel`** for per-user inbox (unread counts, credit balance):

```typescript
import { publishToUserTunnel } from '@/lib/tunnel/publisher'

await publishToUserTunnel(userId, 'notifications:unread', { count: 3 })
```

`lib/discovery/sync-discovery.ts` wraps `publishToChannel` for opportunity and entity CRUD events â€” see [Discovery mutation sync](/en/docs/architecture/discovery-mutation-sync).

## Client subscriptions

Wrap the app in **`TunnelProvider`** (see `AppClientShell`). Components use **`useTunnel()`** for low-level subscribe/publish, or **`useSync()`** to combine REST fetch with tunnel-driven refresh:

```typescript
'use client'

import { useSync } from '@/hooks/use-sync'

const { data, usingTunnel, tunnelConnected } = useSync({
  fetcher: () => fetch('/api/notifications/unread-count').then(r => r.json()),
  tunnel: {
    channel: 'notifications:unread',
    enabled: true,
    onMessage: (message) => ({
      nextData: message.payload as { count: number }
    })
  }
})
```

Chat and typing use `publishToChannel` on the server; clients subscribe to `conversation:${id}` via `useTunnel().subscribe()`.

## Transports (alpha)

| Transport | When used |
|-----------|-----------|
| **Native WebSocket** | `RING_DEPLOY_TARGET` is `k8s` or `self-hosted` â€” primary on custom `server.ts` (`/api/tunnel/ws`) |
| **SSE** | Fallback and primary on `RING_DEPLOY_TARGET=vercel` (serverless) |
| **Long-polling** | Ultimate fallback via `/api/tunnel/poll` |
| **Supabase Realtime** | When `NEXT_PUBLIC_TUNNEL_TRANSPORT=supabase` and Supabase URL/anon key are set |

`TransportManager` picks the first working provider in the configured fallback chain. Tunnel realtime is **orthogonal** to `DB_BACKEND_MODE` â€” see [Backend modes and databases](/en/docs/architecture/backend-modes-and-databases).

## Configuration

```bash
# Deploy target (required) â€” drives primary transport
# Docker image bakes RING_DEPLOY_TARGET=k8s; override in .env.local for dev
RING_DEPLOY_TARGET=k8s                    # vercel | k8s | self-hosted
NEXT_PUBLIC_RING_DEPLOY_TARGET=k8s

# Optional explicit WS URL (default: same host /api/tunnel/ws)
# NEXT_PUBLIC_TUNNEL_WS_URL=wss://your-host/api/tunnel/ws

# Optional explicit client transport override
NEXT_PUBLIC_TUNNEL_TRANSPORT=auto       # auto | sse | websocket | long-polling | supabase

# Supabase Realtime (optional, orthogonal to DB_BACKEND_MODE)
NEXT_PUBLIC_SUPABASE_URL=...
NEXT_PUBLIC_SUPABASE_ANON_KEY=...
```

Verify setup:

```bash
curl "http://localhost:3000/api/tunnel/test"
```

## Channel patterns

| Pattern | Example | Use |
|---------|---------|-----|
| Per-user inbox | `notifications:unread` | Unread badge, credit balance |
| Conversation | `conversation:${id}` | Chat messages, typing |
| Discovery | `opportunities`, `entities` | List freshness after CRUD |
| Matcher | `matcher` | AI matcher moderation events |
| User notify | `user:${userId}` | New conversation alerts |

## Alpha limitations

- **Single-process hub** â€” `InMemoryTunnelHub` does not share state across k8s replicas; horizontal scale requires a shared broker implementation behind `getTunnelHub()`.
- **Channel SSE fan-out** â€” `publishToChannel` delivers to connected SSE sessions; poll clients receive messages for subscribed channels via the poll inbox.
