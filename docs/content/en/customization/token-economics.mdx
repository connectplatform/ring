# Token Economics Setup

<Callout type="info">
  **Ring's philosophy:** "Your platform should run on your economy." Every Ring clone can ship with the reference **RING** token stack or fork it for a branded utility token.
</Callout>

Ring platforms align incentives through on-chain utility: membership fees, marketplace flows, referral rewards, and optional staking. This guide covers:

- The **RING reference tokenomics** shipped in `contracts/RingToken.sol` and `contracts/RingMembership.sol`
- How the **Next.js app** integrates credit balances, subscriptions, and price oracles
- How to **design your own** token for a white-label deployment

## Why token economics matter

**Traditional platforms** centralize value capture — users create content and transactions, but founders and investors retain control.

**Ring token platforms** route fees to a transparent treasury, gate premium features with utility balances, and let communities extend economics through governance and staking over time.

| Flow | RING reference behavior |
|------|-------------------------|
| Membership | 1 RING / month deducted to treasury |
| Minimum buffer | 12 RING (one year) recommended before subscribing |
| Supply | 1,000,000,000 RING minted to treasury at deploy |
| Chain default | Polygon (configurable per clone) |

## RING reference architecture

Ring ships two upgradeable (UUPS) contracts that work together:

<Mermaid>
{`flowchart LR
  User["Member wallet"]
  RT["RingToken\\nERC-20"]
  RM["RingMembership\\nSubscriptions"]
  Treasury["Treasury wallet"]

  User -->|"createSubscription()"| RM
  RM -->|"deductMembershipFee()"| RT
  RT -->|"1 RING / month"| Treasury
  Owner["Owner"] -->|"creditTokens() airdrops"| RT
  RT --> Treasury
`}
</Mermaid>

### RingToken (`contracts/RingToken.sol`)

Upgradeable ERC-20 with burn, pause, and treasury-controlled distribution.

| Parameter | Value | Notes |
|-----------|-------|-------|
| `INITIAL_SUPPLY` | 1,000,000,000 RING | Minted to treasury at `initialize` |
| `MEMBERSHIP_FEE` | 1 RING / month | Only `RingMembership` may call `deductMembershipFee` |
| `MIN_SUBSCRIPTION_BALANCE` | 12 RING | View helper — one year of fees |
| Decimals | 18 | Standard ERC-20 precision |

**Key functions:**

- `deductMembershipFee(member)` — transfers 1 RING from member to treasury; callable only by the linked membership contract
- `creditTokens(recipient, amount, reason)` — owner moves tokens from treasury (airdrops, reimbursements)
- `hasSubscriptionBalance(user)` / `getRemainingMembershipMonths(user)` — UI and ops helpers
- `pause()` / `unpause()` — emergency stop on transfers
- `setTreasury` / `setMembershipContract` — owner configuration after deploy

<Callout type="warning">
  RING is designed as a **utility token** for platform access — not an investment product. Consult counsel before marketing yield, profit expectations, or secondary-market liquidity.
</Callout>

### RingMembership (`contracts/RingMembership.sol`)

Manages recurring membership billing on-chain.

| Parameter | Value |
|-----------|-------|
| `SUBSCRIPTION_PERIOD` | 30 days |
| `GRACE_PERIOD` | 7 days after due date |
| `MAX_FAILED_ATTEMPTS` | 3 before expiry |

**Subscription lifecycle:**

1. `createSubscription()` — requires ≥ 1 RING balance; charges first month immediately
2. `processBatchPayments(batchSize)` — owner/cron processes due renewals
3. Failed payments increment `failedAttempts`; after three failures the subscription moves to `EXPIRED`
4. `cancelSubscription()` — user-initiated; `suspendSubscription` / `reactivateSubscription` — admin
5. `hasActiveMembership(user)` — returns true while `ACTIVE` and within grace window

**Statuses:** `INACTIVE`, `ACTIVE`, `EXPIRED`, `CANCELLED`, `SUSPENDED`

### ReferralRewards (`contracts/contracts-src/ReferralRewards.sol`)

Optional companion contract for **gasless referral payouts**. An operator role pays referrers in the project token using either `MINT` or `TRANSFER` mode, with idempotent `orderRef` tracking to prevent double pays.

Deploy separately when enabling store referral commissions:

```bash
cd contracts && npm run deploy:referral
```

Wire `REFERRAL_REWARDS_ADDRESS` and `REFERRAL_REWARD_TOKEN_ADDRESS` in your clone environment (see [Refcodes](/docs/features/refcodes) — EN canonical for on-chain ops).

## Platform integration

On-chain contracts pair with server-side services so users are not forced to sign every monthly renewal.

### Environment variables

```env
# Smart contracts (Polygon)
NEXT_PUBLIC_RING_TOKEN_ADDRESS=0x...
RING_MEMBERSHIP_CONTRACT_ADDRESS=0x...
RING_TREASURY_WALLET_ADDRESS=0x...
POLYGON_RPC_URL=https://polygon-rpc.com

# Pricing (optional feeds)
CHAINLINK_RING_USD_FEED=0x...
COINMARKETCAP_API_KEY=...

# Referral rewards (optional)
REFERRAL_REWARDS_ADDRESS=0x...
REFERRAL_REWARD_TOKEN_ADDRESS=0x...
```

Client constants live in `constants/web3.ts` — token metadata, staking pool placeholders, and Polygon explorer helpers.

### Credit balance and subscriptions

The app maintains an off-chain **credit ledger** synchronized with membership state:

| Service | Path | Role |
|---------|------|------|
| User credits | `features/wallet/services/user-credit-service.ts` | Balance, spend, membership fee processing |
| Subscriptions | `services/membership/subscription-service.ts` | Create, cancel, renew, batch renewal |
| Price oracle | `services/blockchain/price-oracle-service.ts` | RING/USD with multi-source fallback |

**Typical upgrade flow:**

1. User holds SUBSCRIBER role and sufficient RING credit (≥ 1 token)
2. `subscriptionService.createSubscription` debits credit and records next due date (+30 days)
3. Cron or admin batch job processes renewals; grace period applies before downgrade
4. Role advances to MEMBER on success

**API surface (representative):**

- `GET /api/wallet/credit/balance` — balance + subscription snapshot
- `POST /api/membership/subscription/create` — start subscription
- `POST /api/membership/payment/ring` — process RING payment
- `GET /api/prices/ring-usd` — conversion for UI and receipts

### Staking (optional)

`constants/web3.ts` defines a **RING staking pool** scaffold (`RING_STAKING_ADDRESS`, 20% APR placeholder, 100 RING minimum). Agricultural / cooperative clones may instead wire DAAR/DAARION pools via `features/staking/staking.config.ts`. Treat staking parameters as **per-clone configuration**, not global RING protocol rules.

## Deploying contracts

Hardhat project: `ring-platform.org/contracts/`.

<Steps>
  <Step>
    **Install and compile:**

    ```bash
    cd contracts
    npm install
    npx hardhat compile
    ```
  </Step>

  <Step>
    **Testnet (Polygon Mumbai script):**

    ```bash
  RING_TREASURY_TESTNET=0xYourTreasury \
  npx hardhat run scripts/deploy-testnet.js --network polygon
    ```

    Deploy order: `RingToken` → `RingMembership` → `setMembershipContract`. Deployment JSON is written to `deployments/mumbai-deployment.json`.
  </Step>

  <Step>
    **Mainnet:**

    Requires `RING_TREASURY_MAINNET` and sufficient MATIC on the deployer. Use `scripts/deploy-mainnet.js` only after audit and multisig treasury setup.
  </Step>

  <Step>
    **Wire addresses** into clone `.env`, redeploy the app, and verify contracts on Polygonscan.
  </Step>
</Steps>

<Callout type="warning">
  Run a professional smart-contract audit before mainnet. Use a multisig for treasury and owner keys.
</Callout>

## Designing your own white-label token

Use RING as a template when you need a branded economy.

### Step 1 — Classify the token

<Steps>
  <Step>
    **Utility token (recommended):** Powers listings, membership, or marketplace fees. Avoid profit promises.
  </Step>
  <Step>
    **Security token:** Ownership or revenue share — triggers securities regulation; legal review required.
  </Step>
  <Step>
    **Single vs multi-token:** One token is simpler; split governance and utility only when economics truly require it.
  </Step>
</Steps>

### Step 2 — Set parameters

<Code language="typescript" title="Example clone token config">
{`export const cloneToken = {
  name: 'GreenFood Token',
  symbol: 'GFT',
  totalSupply: 100_000_000,
  decimals: 18,
  membershipFee: 1,           // utility units per month
  minSubscriptionBalance: 12,   // months of runway
  treasuryShare: 1.0,           // 100% minted to treasury at launch
}`}
</Code>

**Distribution patterns:**

- **Community-first:** 40–50% rewards and airdrops, 20% staking, 20% development (vested), 10% treasury
- **Enterprise:** Higher treasury and partner allocations; limited on-chain governance

**Supply policy:** Fixed supply (RING model), controlled inflation (staking rewards), or deflationary burns on marketplace fees.

### Step 3 — Map fees to features

<Code language="typescript" title="Fee structure example">
{`export const feeStructure = {
  opportunityListing: 10,
  marketplaceTransactionBps: 100, // 1%
  premiumFeaturesPerMonth: 100,
  vendorMinStake: 50,
}`}
</Code>

Collect fees to treasury; route a portion to staking rewards or burns according to your tokenomics paper.

### Step 4 — Integrate with Ring

1. Fork `RingToken.sol` / `RingMembership.sol` or deploy as-is with your treasury
2. Update `constants/web3.ts` and env addresses
3. Point `user-credit-service` and `subscription-service` at your fee constants
4. Enable wallet UI and membership gates in portal config

## Economic model

<Mermaid title="Token fee flow">
{`flowchart LR
  Fees["User fees<br/>• Membership<br/>• Listings<br/>• Marketplace"]
  Treasury["Treasury<br/>(on-chain)"]
  Ops["Ops & growth<br/>• Development<br/>• Rewards pool<br/>• Referrals"]
  Gov["Governance<br/>(future / clone)"]

  Fees --> Treasury --> Ops
  Treasury --> Gov`}
</Mermaid>

**Health metrics to monitor:**

- Active subscriptions and failed payment rate
- Treasury balance and credit ledger reconciliation
- Token velocity and holder concentration
- RING/USD oracle latency and fallback usage

## Legal and documentation

<Callout type="warning">
  Token launches have jurisdiction-specific consequences. Engage legal counsel before public sale, exchange listing, or yield marketing.
</Callout>

- Publish a clear **tokenomics paper** — utility, fees, treasury, no investment claims
- Maintain **audit reports** and upgrade admin key procedures
- Disclose volatility and tax obligations to users
- Consider KYC/AML for large fiat on-ramps
