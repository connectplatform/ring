## 🚀 Implementation & Integration

Ring Platform v1.6+ stores opportunities in **PostgreSQL via `DatabaseService`** (default `DB_BACKEND_MODE=k8s-postgres-fcm`). Server services return **`SerializedOpportunity`** (ISO date strings) for App Router and client components. Legacy Firestore converters under `lib/converters/` apply only when `DB_BACKEND_MODE=firebase-full`.

### Data layer (read vs write)

| Path | Module | When |
|------|--------|------|
| **PostgreSQL read/write** | `features/opportunities/lib/opportunity-db-mapper.ts` | Default production clones — maps JSONB rows to `SerializedOpportunity` |
| **Post-mutation sync** | `features/opportunities/lib/opportunity-mutation-sync.ts` | After create/update/delete — `revalidateTag`, `revalidatePath`, Tunnel publish |
| **My Opportunities** | `/[locale]/opportunities/my` | Canonical dashboard (`ROUTES.MY_OPPORTUNITIES`); legacy `/opportunities/my-opportunities` removed |
| **Firestore legacy** | `lib/converters/opportunity-converter.ts` | `firebase-full` only — `FirestoreDataConverter` |
| **Client transport** | `lib/converters/opportunity-serializer.ts` | `Timestamp` ↔ ISO when bridging Firestore-shaped objects |

### Matcher auto-approval (admin)

When **Platform Settings → AI → Matcher → Auto-approve** is enabled, new opportunities in `pending` status can be promoted to `active` after LLM matching passes score and confidence gates (`features/opportunities/services/auto-approval-service.ts`). Configure defaults in `ring-config.json` → `matcher` and override per tenant in `platform_settings`.
